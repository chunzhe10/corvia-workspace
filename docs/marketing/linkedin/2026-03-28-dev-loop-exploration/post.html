The most valuable part of AI-assisted development isn't writing code. It's the thinking before the code. My workflow has two layers. Layer 1 is creative. I explore a vague problem with Claude, brainstorm approaches, write an RFC, and run a 5-persona review on the design. The output: a structured issue board with dependency graphs, acceptance criteria, and tracking issues. "Knowledge should have a lifecycle" became 13 structured GitHub issues across 5 phases. "Search recall is 37.5%" became 7 issues with measurable ship gates. Layer 2 is execution. The dev-loop is a skill file that Claude Code reads when I type `/dev-loop 14`. It orchestrates the full pipeline: fetches the GitHub issue, validates the design, breaks work into tasks, implements with subagents, runs 5-persona review on the code, and opens a PR. For the knowledge lifecycle feature, all 13 issues went through this. 13 PRs merged. Zero design rewrites. I persist all the research with corvia (github.com/chunzhe10/corvia). When I revisit a topic, I see what I explored before and what changed since. The knowledge compounds instead of evaporating. Layer 1 is where I learn. Layer 2 is where Claude excels. Skipping Layer 1 is how you end up rewriting things three times. Do you have a structured exploration phase, or do you jump straight to code? #AIAssistedDev #DevWorkflow #ClaudeCode #BuildInPublic #DesignFirst
