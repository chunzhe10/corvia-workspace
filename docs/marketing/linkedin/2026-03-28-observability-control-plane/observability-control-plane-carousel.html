The Problem
The Agent Builds It.
But Can't See It.
I'm building an organizational memory system in Rust. An AI agent
(Claude Code) writes the code, runs the tests, ships the commits.
But when something is slow or broken, it's flying blind.
What the agent couldn't answer
❌
"Why is search taking 120ms?"
Is it embedding? Vector search? Both? No way to tell without manual profiling.
❌
"Are there stale sessions accumulating?"
Agent sessions can go orphaned. No visibility into lifecycle state.
❌
"How many entries are in the knowledge store?"
The agent writes to the system constantly. No dashboard, no counters.

The agent is both the builder and the first user. If it can't observe the system, it's debugging with println.

Retrieval consistently takes 60-85% of RAG pipeline time. That insight redirected optimization from the embedding model to the vector search path.

The Next Question
It Can See.
But Can It Act?
Observability showed the agent what was happening.
But it had no way to change anything without running shell commands.
What I wanted to do but couldn't
📊
Tune RAG parameters live
Change top_k, similarity cutoff, graph expansion depth. Hot-reload. See the impact on the next query.
🔧
Rebuild the vector index
After a schema change or chunking update. No SSH, no CLI. Tell the agent to do it.
🔄
Toggle features and A/B test
Enable skills, change chunking strategy, adjust merge thresholds. All hot-reloadable.

Observability without control is a dashboard with no buttons. I want to operate the system through the agent, with the same protocol it already speaks.

Step 2: Control Plane
10 MCP Tools.
3 Safety Tiers.
Same MCP server. Same protocol the agent already uses for knowledge.
Now it can also manage the system. With graduated trust.
Tier 1: Auto-Approved (5 tools)
system_status
config_get
agents_list
adapters_list
merge_queue
Tier 2: Confirm First (3 tools)
config_set
gc_run
rebuild_index
Tier 3: Confirm + Dry-Run (2 tools)
agent_suspend
merge_retry

Built on MCP's existing primitives (tool annotations, _meta). Read freely. Mutate with a preview. Destructive actions need both a dry-run and a confirmation.

What This Leads To
The Product Becomes
A Lab Bench.
Observe, hypothesize, change, measure. All through MCP.
All without leaving the conversation.
1
Observe
Traces show retrieval is 60-85% of RAG latency. Graph expansion adds significant cost.
2
Hypothesize
"What if I disable graph expansion and increase top_k instead?"
3
Act
config_set rag.graph_expansion_depth = 0. Hot-reload. No restart.
4
Measure
Run the same queries. Read the traces. Compare p50 and p95 before and after.

Same loop works for A/B testing skills, hot-swapping embedding models, changing chunking strategies. The agent is both the experimenter and the instrument.

Key Takeaways
What I've
Learned So Far
💡
Instrument at design time, not after the fire
Span contracts defined before the code existed. Every new feature gets traces automatically.
👁
Observability without control is a dashboard with no buttons
The agent needs graduated permission to act on what it sees. Not just watch.
🔒
Safety tiers are the agent equivalent of "are you sure?"
Built on MCP's existing primitives. Read freely. Mutate with preview. Destructive ops need dry-run.
🌱
I'm still early. The tools are shipped. The workflows are forming.
MCP standardized tool access. Agent control planes are next. The patterns exist. Someone needs to make them plug-and-play.
